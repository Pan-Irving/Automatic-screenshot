影刀 DeepSeek 采集 MVP 实现说明
生成时间：2026-06-02

一、项目目标

这个 MVP 的目标是用影刀作为任务编排入口，用真实 Chrome 和真实 DeepSeek 登录态完成问题采集，降低原先 Playwright 自动化方案触发 DeepSeek 验证/风控的概率。

第一版只支持 DeepSeek，不接入现有 FastAPI/Web UI。任务从本地 Excel 读取，采集结果写回 Excel，并在本地 result 目录保存截图、回答文本和对话链接。

核心要求：

1. 每条 Excel 数据对应一个独立 DeepSeek 新对话，避免上下文污染。
2. 不绕过验证码，不自动过验证；遇到登录、验证、风控时标记人工处理。
3. 使用真实 Chrome 页面采集，不生成假截图，不用文本重绘代替页面证据。
4. 截图尽量包含完整问题和完整回答。
5. 每张截图上叠加采集时间戳。
6. 保存 DeepSeek 当前对话链接到 Excel，并在 result 文件夹里额外生成链接 txt。

三、用到的技术

1. 影刀 RPA

影刀负责触发本地 Python 流程、读取 Excel、写回结果。当前没有依赖影刀的网页元素捕获能力作为主采集方式，因为网页元素捕获和坐标点击不稳定，容易出现输入框点不到、截图只截视口等问题。

2. Python

collector.py 是主要执行逻辑。主要职责：

- 读取 questions.xlsx。
- 按行筛选 pending 且 platform=deepseek 的任务。
- 写回 running、success、failed、manual_required 等状态。
- 调用 Chrome CDP 提交问题、等待回答、截图。
- 保存 .png、.txt、_url.txt。

3. openpyxl

用于读写 Excel：

- 读取任务列。
- 自动补充 answer_url 列。
- 写回状态、截图路径、回答文本路径、对话链接、备注、更新时间。

4. Google Chrome CDP

CDP 是 Chrome DevTools Protocol。当前方案通过本地 127.0.0.1:9222 端口控制真实 Chrome 页面。

使用 CDP 的原因：

- 保留真实 Chrome、真实账号、真实登录态。
- 不依赖 Playwright 启动自动化浏览器，减少自动化特征。
- 不需要开启 Chrome 的“允许 Apple 事件中的 JavaScript”。
- 不依赖鼠标坐标点击，输入和截图更稳定。

当前使用到的 CDP 能力：

- /json/list：获取当前标签页列表。
- /json/new：新建 DeepSeek 标签页。
- /json/close/{target_id}：关闭旧 DeepSeek 标签页。
- Runtime.evaluate：在页面中执行 JS，读取文本、定位输入框、判断页面状态、注入截图样式。
- Input.insertText：向 DeepSeek 输入框插入问题。
- Input.dispatchKeyEvent：发送 Enter 兜底。
- Page.captureScreenshot：截取真实页面图像。
- Page.getLayoutMetrics：读取页面尺寸。

5. Pillow

用于图片后处理：

- 滚动截图分段拼接。
- 给截图叠加“采集时间：YYYY-MM-DD HH:MM:SS”时间戳。
- 计算两段截图的相似度，用于裁剪重叠区域。

6. macOS Shell / AppleScript

start_chrome_cdp.command 使用 shell 启动 Chrome。

少量 AppleScript 仍保留在旧桌面兜底函数里，但当前主流程已改成 CDP-only。正常情况下不再依赖 AppleScript 点击输入框或执行页面 JS。

所以这里使用 chrome_cdp_profile 保存专用登录态。第一次启动后，需要在这个专用 Chrome 中手动登录 DeepSeek。后续会复用该登录态。

七、主流程

主入口逻辑：

1. 打开 questions.xlsx。
2. 查找 status=pending 且 platform=deepseek 的行。
3. 每次只处理一行，不并发。
4. 将该行写成 running。
5. 调用 run_deepseek。
6. 写回采集结果。
7. 任务之间等待 TASK_DELAY_SECONDS + 随机延迟。

当前延迟参数：

- TASK_DELAY_SECONDS = 10
- TASK_DELAY_RANDOM_SECONDS = 10

这意味着每题之间大约等待 10-20 秒。早期方案是 60-180 秒，后来为了调试和效率调低了。


八、每题独立新对话

每条任务都会强制创建独立 DeepSeek 新对话，避免上下文污染。

实现方式：

1. 调用 _cdp_close_deepseek_pages。
2. 关闭所有当前已打开的 chat.deepseek.com 页面。
3. 调用 _cdp_new_deepseek_page。
4. 新建一个新的 https://chat.deepseek.com/ 标签页。
5. 在新页面中提交当前 Excel 行的问题。

这样不会复用旧的 /a/chat/s/... 页面，也不会把上一条问题的上下文带入下一条问题。


九、登录/验证/风控处理

脚本不会绕过验证码或自动过验证。

检测方式：

- URL 包含 /sign 或 /login 时，认为需要登录。
- 页面文本包含验证码、安全验证、安全检测、人机验证、请完成验证、访问受限、captcha、verify 等关键词时，认为需要人工处理。

如果触发：

1. 保存当前页面截图。
2. 截图上加采集时间戳。
3. Excel 写回 manual_required。
4. remark 记录原因。

注意：

早期版本曾经把任意“验证”二字都当成风控，导致历史对话标题里包含“截图验证问题”时误判。现在已收窄为“安全验证”“请完成验证”等更明确的词。


十、问题提交方式

当前不使用鼠标坐标，不使用剪贴板粘贴作为主流程。

提交步骤：

1. CDP Runtime.evaluate 找到输入框：
   textarea、contenteditable、role=textbox、ProseMirror 等。
2. 清空输入框。
3. 使用 CDP Input.insertText 插入问题文本。
4. 查找发送按钮并点击。
5. 如果点击发送失败，使用 Enter 事件兜底。

这样比固定坐标点击稳定很多，也不会受 Chrome 窗口大小变化影响。


十一、等待回答完成

等待逻辑不是固定 sleep。

脚本会轮询：

- 页面中是否存在“停止生成 / stop generating”等按钮。
- 当前回答文本是否还在变化。
- 回答文本是否稳定。

结束条件：

- 有回答文本。
- 不再生成。
- 文本稳定一段时间。

超时参数：

- ANSWER_TIMEOUT_SECONDS = 150

如果超时但已经拿到部分文本，会尽量保存已有文本，并在 remark 中体现。


十二、回答文本保存

回答文本通过页面 DOM 提取。

选择器大致包括：

- .ds-markdown
- class 中包含 markdown 的元素
- article
- class 中包含 answer 或 message-content 的元素

提取后写入：

{id}_{platform}_round{round}_{HHMMSS}.txt

文本文件使用 UTF-8。


十三、对话链接保存

每次提交并等待回答后，会读取当前 DeepSeek 页面 URL。

链接保存到两个位置：

1. Excel answer_url 列。
2. result 文件夹里的 _url.txt。

示例内容：

https://chat.deepseek.com/a/chat/s/xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx

这样即使之后只看本地结果文件夹，也可以知道截图和回答对应哪个 DeepSeek 对话页面。


十四、截图总体策略

截图必须是真实 DeepSeek 页面截图，不接受用辅助功能文本或 PIL 重画成“证据图”。

当前截图有两种路径：

1. clean QA DOM 截图

脚本会临时克隆当前问题和回答相关 DOM，形成一个干净的 capture root，再用 CDP 截图。

优点：

- 画面干净。
- 不容易包含侧边栏和输入框。

缺点：

- DeepSeek 某些回答结构复杂时，克隆节点可能继承高度/overflow 限制，导致只截到顶部。

因此当前策略是：

- 很短回答可以用 clean QA DOM。
- 只要回答超过较短阈值，就优先走滚动拼接。

2. CDP 真实页面滚动拼接

这是当前常规回答和长回答的主要截图方式。

流程：

1. 进入 QA capture mode。
2. 临时隐藏侧边栏、底部输入框、固定浮层、滚动条装饰。
3. 定位聊天滚动容器。
4. 按滚动位置分段调用 Page.captureScreenshot。
5. 使用 Pillow 拼成长图。
6. 给最终图片加时间戳。
7. 退出 QA capture mode，恢复页面。


十五、QA capture mode

QA capture mode 是为了让截图只保留问答主体，减少页面固定元素干扰。

进入 capture mode 时会临时：

- 隐藏左侧历史栏。
- 隐藏底部输入框。
- 隐藏 fixed/sticky 浮层。
- 隐藏滚动条装饰。
- 保持页面滚动行为为 auto。

退出 capture mode 时会：

- 移除临时 style。
- 移除临时隐藏标记。
- 移除滚动容器标记。
- 恢复原页面。

注意：

这个过程只是临时调整页面 CSS/DOM 显示状态，用于截图。底层内容仍是 DeepSeek 真实页面，不是生成假图。


十六、滚动拼接逻辑

早期问题：

- 如果每次按一整屏高度跳转，DeepSeek 的虚拟滚动/固定区域会导致中间内容漏掉。
- 如果重叠过大，截图会出现大段重复。

当前策略：

1. 分段截图时保留一定重叠。
2. 用 Pillow 对上一张和下一张做图像相似度比较。
3. 自动裁剪重叠部分后拼接。
4. 相似度阈值当前设置为 5。

阈值说明：

- 数值越小，越严格，越不容易把不相似区域当成重叠。
- 数值越大，越容易裁掉更多重复，但也更可能误裁内容。

当前用户偏好是“重复少一点，可以稍微极限一点”，所以阈值已调为 5。

仍需注意：

DeepSeek 页面是动态页面，表格、引用角标、虚拟滚动、固定输入框都会影响拼接。滚动拼接已经比单屏截图完整很多，但不是像浏览器原生 fullPage screenshot 那样完全稳定。


十七、时间戳

每张截图保存后都会叠加时间戳。

格式：

采集时间：YYYY-MM-DD HH:MM:SS

位置：

图片右上角。

实现方式：

- 使用 Pillow 打开最终 PNG。
- 使用中文字体绘制文字。
- 添加半透明白色背景框，避免和网页内容混在一起。
- 保存回原 PNG。

适用范围：

- 成功截图。
- 失败截图。
- manual_required 截图。
- CDP 页面截图。
- 滚动拼接截图。
- 桌面兜底截图。


十八、错会话保护

截图前会检查当前 DeepSeek 页面是否包含当前 Excel 的问题文本。

原因：

调试过程中发现 CDP 浏览器里可能同时有 DeepSeek、Gemini、ChatGPT 等多个标签。如果没有严格校验，可能把别的页面或历史对话截图覆盖到当前任务文件中。

当前保护：

- _cdp_deepseek_ws_url 只会返回 chat.deepseek.com 标签页，不会退回任意普通 page。
- 截图前会检查页面文本中是否包含当前 question 的前缀。
- 如果不匹配，会返回 question_not_found_on_page，避免错误覆盖。


十九、失败处理

常见失败与处理：

1. CDP 不可用

原因：

- 没运行 start_chrome_cdp.command。
- Chrome 不是用 9222 端口启动。
- 端口被关闭。

处理：

- Excel 写 failed。
- remark 提示需要运行 start_chrome_cdp.command。

2. 登录/验证/风控

处理：

- 截图保存。
- Excel 写 manual_required。
- 人工处理后把该行 status 改回 pending 再跑。

3. 输入框或发送按钮找不到

处理：

- 截图保存。
- Excel 写 failed。
- remark 写 cdp_input_not_found_or_send_failed。

4. 截图失败

处理：

- 尽量保存页面截图。
- 回答文本如果已拿到，仍会写入 txt。
- Excel 写 failed。
- remark 写 cdp_screenshot_failed。


二十、当前已知问题

1. DeepSeek 长回答滚动拼接仍可能出现少量重复或边界不够自然。

原因：

- DeepSeek 页面不是普通静态长页面。
- 有虚拟滚动、固定输入框、动态表格和引用角标。
- Page.captureScreenshot 每段截的是当前视口状态，拼接需要靠图像相似度估算。

当前缓解：

- QA capture mode 隐藏无关固定元素。
- 分段截图带重叠。
- 相似度阈值设置为 5，尽量减少重复。

2. 如果回答中有很宽的表格，截图可能仍会保留较宽画布。

原因：

- 为了不裁掉表格右侧内容，裁剪区域需要保守。

3. 如果 DeepSeek 页面结构大改，选择器可能需要更新。

受影响位置：

- 输入框定位。
- 回答文本提取。
- 滚动容器识别。
- 侧边栏/输入框隐藏规则。

二十三、当前实现总结

这个 MVP 的核心思路是：

影刀负责本地任务编排，Chrome CDP 负责真实页面控制，Excel 负责任务和状态交换，本地文件夹负责证据落盘。

相比原 Playwright 方案：

- 更接近真实用户 Chrome 登录态。
- 不依赖自动化浏览器上下文。
- 仍能保留较完整的截图、回答文本和对话链接。

相比纯影刀网页元素/坐标方案：

- 输入更稳定。
- 截图能力更强。
